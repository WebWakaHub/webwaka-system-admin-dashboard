## SECTION II — CANONICAL FINANCE DOMAIN SCOPE

The **Finance (FIN)** domain is the exclusive authority for all monetary value, financial transactions, and ledger integrity within the WebWaka platform. It operates on a **Ledger-Centric Model**, where the immutable, double-entry ledger is the source of truth. The `FIN` domain records the financial consequences of business events that occur in other domains but is not responsible for the business logic of those events themselves.

### Inclusions: What Finance (FIN) Governs

| Capability | Description |
| :--- | :--- |
| **Ledger Management** | Maintains the immutable, double-entry ledger, ensuring all financial transactions are recorded and balanced. This is the core of the `FIN` domain. |
| **Transaction Recording** | Processes and records all financial events, such as payments, refunds, and transfers, as atomic transactions in the ledger. |
| **Settlement & Payout Orchestration** | Manages the process of settling funds, calculating vendor commissions, and orchestrating payouts to external accounts. |
| **Commission & Tax Logic** | Contains the rules and logic for calculating platform commissions and applying relevant taxes to financial transactions. |
| **Subscription Billing (Monetary)** | Manages the financial aspect of subscriptions, such as creating invoices and recording recurring payments. The *entitlement* aspect (granting access) is handled separately. |
| **Refund Finalization** | Processes the monetary side of a refund, debiting the appropriate accounts. The business decision to initiate a refund comes from the `COM` domain. |
| **Reconciliation Workflows** | Provides tools and processes for reconciling internal ledgers with external statements from payment gateways and banks. |
| **Financial Reporting Primitives** | Exposes the foundational data and queries required to build financial reports, such as profit and loss statements or balance sheets. |

### Exclusions: What Finance (FIN) Does NOT Govern

To maintain strict domain isolation and constitutional integrity, the following capabilities are explicitly excluded from the `FIN` domain:

| Capability | Canonical Domain | Rationale |
| :--- | :--- | :--- |
| **Business Logic of Commerce** | `COM` (Commerce) | The `FIN` domain does not understand products, shopping carts, or order workflows. It only reacts to financial events generated by the `COM` domain (e.g., `payment_required_for_order`). |
| **Entitlement Gating** | `Entitlement Model` | The `FIN` domain records that a subscription has been paid, but it does not grant the user access to the corresponding features. This is the responsibility of the cross-cutting Entitlement system. |
| **Logistics & Fulfillment** | `LOG` (Logistics & Fulfillment) | The cost of shipping is a financial data point, but the `FIN` domain is not involved in the physical logistics of moving goods. |
| **Identity & Authentication** | `IDA` (Identity & Access) | The `FIN` domain associates transactions with a user ID, but it does not manage user identity, authentication, or authorization. |
| **Payment Gateway Binding** | `Runtime Plane` | The `FIN` domain operates on abstract payment events (e.g., `payment.succeeded`). The specific implementation of a payment gateway (e.g., Paystack, Stripe) is a runtime concern, handled by adapters in the Runtime Plane. |

## SECTION V — LEDGER MODEL DEFINITION

The canonical **Finance (FIN)** domain is founded on a **Ledger-Centric Architecture**. The ledger is the immutable, single source of truth for all financial state. The following principles are constitutionally mandated.

| Principle | Definition |
| :--- | :--- |
| **Double-Entry Invariant** | Every financial transaction must be recorded as a set of debits and credits across at least two accounts. For every transaction, the total value of debits must equal the total value of credits (`Σ Debits = Σ Credits`). This ensures the ledger remains balanced and that value is never created or destroyed, only moved. |
| **Transaction Atomicity** | All entries (debits and credits) for a single financial transaction must be committed to the ledger together, or not at all. The ledger must never be left in a partially updated, inconsistent state. This is an absolute, non-negotiable requirement. |
| **Idempotency** | Financial operations must be idempotent. Processing the same financial event (e.g., a `payment.succeeded` webhook) multiple times must result in the same final state as processing it once. This is typically achieved by tracking processed event IDs. |
| **Event-Sourced vs. State-Stored** | While the ledger may be implemented using an event-sourcing pattern internally, the canonical state of an account's balance must be readily available as a stored value for performance. The stored state is a projection of the event stream and must be reconcilable against it. |
| **Commission & Settlement Modeling** | Commissions and settlements are not direct payments. They are modeled as internal ledger transactions. For example, a sale results in a credit to a temporary holding account. A separate settlement transaction then debits the holding account and credits the vendor's wallet and the platform's commission account. |
| **Refund Invariant** | A refund is not a deletion of a past transaction. It is a new, separate transaction that reverses the flow of value. A refund transaction debits the recipient's account and credits the payer's account, with its own unique transaction ID and audit trail. |
| **Tax Calculation Abstraction** | Tax calculation logic must be an abstract component that can be applied to a transaction. The ledger records the tax amount as a separate line item in the transaction, crediting a dedicated tax liability account. |

### Foundational Declaration

**Ledger integrity must not, under any circumstances, depend on the state or validity of data from the Commerce (COM) domain or any other non-FIN domain.** The `FIN` domain trusts only its own internal state and the verified, idempotent events it consumes.

## SECTION VI — PAYMENT ADAPTER ISOLATION ANALYSIS

The legacy `payment-billing` module demonstrates a critical architectural flaw: the direct integration of payment gateway logic (Paystack, Flutterwave) within the application's core. This creates tight coupling to specific vendors and violates the `STRICT_INFRASTRUCTURE_NEUTRAL_IMPLEMENTATION_CONTRACT.md`.

### Analysis of Legacy Implementation

*   **Direct SDK Coupling:** The legacy code directly imports and uses SDKs or APIs from payment providers like Paystack and Flutterwave within its services.
*   **Hard-Coded Flows:** The logic for handling payment initialization, webhooks, and verification is often tailored to the specific API contracts of a single provider.
*   **Configuration in Application Code:** API keys and other sensitive provider-specific configurations are managed within the application's environment, blurring the line between application logic and infrastructure binding.

### Canonical Resolution: The Runtime Plane Adapter Model

The canonical architecture strictly forbids this coupling. The following rule is absolute:

> **Payment gateways are infrastructure. All direct interaction with payment gateway APIs and SDKs belongs exclusively to Adapters within the Runtime Plane.**

The `FIN` domain must remain completely ignorant of which payment gateway is being used, or if one is being used at all. The interaction model is as follows:

1.  **`FIN` Emits an Abstract Event:** When a payment needs to be processed, the `FIN` domain emits a generic, provider-agnostic event, such as `payment_processing_requested`, containing only the necessary financial data (amount, currency, reference ID).
2.  **Runtime Adapter Receives the Event:** A specialized **Payment Gateway Adapter** in the Runtime Plane subscribes to this event.
3.  **Adapter Interacts with the Gateway:** This adapter contains the direct SDK/API integration. It translates the abstract event into a specific API call to the payment gateway (e.g., Paystack).
4.  **Adapter Emits a Standardized Response Event:** After the gateway interaction is complete, the adapter consumes the provider-specific response and translates it back into a standardized, provider-agnostic event, such as `payment_succeeded` or `payment_failed`. This event includes a verified status and gateway reference.
5.  **`FIN` Consumes the Standardized Event:** The `FIN` domain subscribes to these standardized response events and updates its internal ledger accordingly.

This model ensures that the entire biological architecture of the `FIN` domain can operate without any modification, whether payments are processed by Paystack, Stripe, bank transfers, or even internal test harnesses. The choice of payment provider is a runtime binding, not a biological dependency.

## SECTION VII — COM ↔ FIN BOUNDARY ENFORCEMENT

The boundary between the Commerce (`COM`) and Finance (`FIN`) domains is the most critical in the platform for preventing architectural collapse. This section explicitly defines the contract and sequence of events that enforce this separation.

| Interaction | `COM` Domain Responsibility | `FIN` Domain Responsibility | Event Contract |
| :--- | :--- | :--- | :--- |
| **Order Placement** | The `OrderManager` organ orchestrates the creation of an order based on the contents of a `ShoppingCart`. It calculates the total amount due. | The `FIN` domain is not involved at this stage. | `COM` emits `order.created` with the total amount. `FIN` does not listen to this. |
| **Payment Initialization** | The `CheckoutOrchestrator` organ determines that a payment is required to confirm the order. | The `FIN` domain receives a `payment_authorization_requested` event. It creates a `pending` transaction in its ledger and interacts with the Runtime Plane to get a payment URL. | `COM` emits `payment_authorization_requested`. `FIN` emits `payment_authorization_provided` with the payment URL. |
| **Payment Completion** | The `COM` domain is not directly involved. It waits for a `payment.succeeded` event. | The `FIN` domain receives a standardized `payment.succeeded` event from the Runtime Plane Adapter. It marks the corresponding transaction as `completed` in its ledger. | `Runtime Adapter` emits `payment.succeeded`. `FIN` consumes this and then emits `financial_transaction.completed`. |
| **Order Confirmation** | The `OrderManager` consumes the `financial_transaction.completed` event and, upon its receipt, confirms the order and begins the fulfillment process (e.g., by emitting an event for the `LOG` domain). | The `FIN` domain's primary role in this step is complete. Its ledger now accurately reflects the completed payment. | `FIN` emits `financial_transaction.completed`. `COM` consumes this. |
| **Refund Initiation** | A business process within `COM` (e.g., a customer service action) decides a refund is necessary. | The `FIN` domain is not involved in the business decision to refund. | `COM` emits `refund.initiated` with the order and amount details. |
| **Refund Processing** | The `COM` domain is not involved in the monetary transaction. | The `FIN` domain consumes the `refund.initiated` event. It creates a new, separate refund transaction in its ledger, reversing the flow of value. | `FIN` consumes `refund.initiated` and emits `financial_transaction.refunded` upon completion. |

This event-driven, contract-based interaction ensures that neither domain needs to know about the internal workings of the other. `COM` manages the 
business process of a sale; `FIN` manages the immutable financial record of that sale. Their states are independent, and their interaction is strictly asynchronous and contract-based.

## SECTION IX — FEDERATION COMPATIBILITY ANALYSIS

The canonical `FIN` domain must be designed for federation, allowing for financial consistency and interoperability across multiple, independent WebWaka instances. This requires a strict discipline regarding ledger management and versioning.

| Federation Concern | Canonical Declaration |
| :--- | :--- |
| **Ledger Version Upgrade Discipline** | The schema and validation rules of the canonical `Ledger` organ are subject to the `VERSION_EVOLUTION_AND_PATCH_GOVERNANCE_MODEL.md`. Any change to the ledger's fundamental structure is a major version change and requires a formal, platform-wide migration strategy. No single instance can unilaterally alter its ledger schema. |
| **Cross-Instance Patch Propagation** | Patches to the `FIN` domain's logic (e.g., a bug fix in the `CommissionCalculator`) must be propagated to all federated instances simultaneously to ensure consistent financial calculations across the federation. This is enforced by the Federation Plane's governance protocols. |
| **Settlement Consistency Guarantees** | In a federated system where a transaction on one instance may result in a payout on another, the settlement process must be managed by a higher-level Federation Plane service. This service will orchestrate cross-instance ledger transactions to ensure atomicity and consistency. |
| **Backward Compatibility Constraints** | All changes to the `FIN` domain's event contracts (the structure of events like `financial_transaction.completed`) must be backward-compatible for at least one major version. This prevents a single upgraded instance from breaking financial data consumption for other, older instances in the federation. |
