## 1. Purpose of This Document

WebWaka exists to be a **platform-for-platforms**—an extensible digital operating system that empowers entrepreneurs and organizations across Africa to build and scale complex, multi-sided marketplaces and transactional ecosystems.

### 1.1. What WebWaka Is

WebWaka is **not a single product**. It is a foundational layer—a set of composable primitives, suites, and infrastructure—upon which a universe of specialized platforms can be built. It is the operating system for digital commerce, logistics, social interaction, and service delivery in emerging markets.

### 1.2. What WebWaka Is Not

- **Not a SaaS tool** (like GoHighLevel or Skool)
- **Not a marketplace** (like Shopify or Amazon)
- **Not a ride-hailing app** (like Uber or InDrive)
- **Not a CRM** (like Salesforce or HubSpot)

WebWaka is the platform that enables others to build all of the above—and more.

### 1.3. Why WebWaka Must Exceed Existing Platforms

Existing platforms (GoHighLevel, Skool, InDrive, Shopify) offer fixed feature sets. WebWaka provides **structural superiority** through:

1. **Recursive Platform Design:** WebWaka uses itself to build and manage itself
2. **Plugin-First Architecture:** All features are plugins to a minimal kernel
3. **Event-Driven by Default:** All state changes emit events for real-time capabilities
4. **Offline-First as Non-Negotiable:** All systems function in low-connectivity environments
5. **AI-Native, Not AI-Added:** AI is horizontal and permission-driven, not a feature bolt-on

### 1.4. The African Market Context

WebWaka is designed for Africa-first constraints:
- **Low connectivity:** Offline-first is mandatory
- **Mobile-first:** Most users access via smartphones
- **Cash-heavy economies:** Support for cash transactions and agent banking
- **Multi-language:** Support for English, French, Swahili, Hausa, Yoruba, and more
- **Regulatory complexity:** Compliance with diverse national regulations

## 2A. Environmental Reality Invariants (Mobile-First, PWA-First, Offline-First, Africa-First)

### Purpose

This section defines the non-negotiable environmental realities under which WebWaka is designed, built, and operated. These realities are not implementation preferences; they are hard constraints derived from real-world operating conditions in Nigeria and across Africa.

All architecture, product design, engineering decisions, governance policies, and CI enforcement MUST assume these constraints by default.

Failure to comply constitutes a violation of the WebWaka canonical context.

---

### 2A.1 Mobile-First Is Canonical (Not Responsive-First)

Mobile is the primary execution environment for WebWaka.
- All features MUST be designed mobile-first, not desktop-first.
- Desktop and tablet experiences are secondary adaptations.
- Touch interaction, small screens, low memory devices, and intermittent usage are the baseline.
- No feature may be designed assuming keyboard, mouse, or large-screen availability.

Responsive design alone is insufficient.
Design must be mobile-native in behavior, not merely layout.

---

### 2A.2 PWA-First Is Mandatory

All end-user applications MUST be delivered as Progressive Web Applications (PWA) by default.
- Offline caching is mandatory.
- Installability is mandatory.
- Background sync is mandatory where applicable.
- PWA functionality MUST be meaningful on low-end Android devices.

Native mobile applications MAY exist, but ONLY as optional accelerators — never as the primary delivery mechanism.

---

### 2A.3 Offline-First Is Structural, Not Cosmetic

Offline-first is a core architectural property, not a UX enhancement.

All systems MUST assume:
- Prolonged offline operation
- Partial sync success
- Application termination without warning
- Network flapping
- Duplicate event submission

Offline-first explicitly requires:
- Local persistence of transactions
- Local persistence of inventory state
- Local persistence of identity and authorization context
- Deterministic sync and reconciliation

Read-only offline modes are not acceptable.

This invariant is enforced via:
- FD-2026-002 (Transaction Queue Persistence)
- FD-2026-003 (Sync-On-Reconnect)

---

### 2A.4 Intermittent Power and Connectivity Assumption

The platform MUST assume:
- Sudden power loss
- Battery exhaustion
- Device restarts during active workflows
- Long gaps between successful sync windows

No workflow may assume uninterrupted execution.

Durability across crashes and power loss is mandatory.

---

### 2A.5 Nigeria-First, Africa-First Execution Context

WebWaka is Nigeria-first by design and Africa-first by extension.

Global scalability MUST NOT dilute local operability.

Design assumptions MUST prioritize:
- Low-cost Android devices
- Inconsistent bandwidth
- High latency
- Cost sensitivity
- Field-based usage (markets, motor parks, retail floors)

Any design that works globally but fails locally is invalid.

### 2F. Field Operations Reality Layer (Africa-First Execution)

WebWaka is designed for environments where infrastructure constraints are the default, not the exception. The platform MUST explicitly account for the realities of field operations in Nigeria and across Africa, including but not limited to intermittent connectivity, shared devices, low-end hardware, human-mediated trust, and variable digital literacy.

#### Field Reality Invariants

1. **Shared Device Assumption**
   The platform MUST assume that multiple humans may operate from a single physical device.
   Role switching, session isolation, and audit attribution MUST function offline.

2. **Device Volatility Tolerance**
   The platform MUST tolerate:
   - Device restarts
   - Battery loss
   - Device theft or replacement
   without catastrophic data loss or role corruption.

3. **Offline Identity Continuity**
   Users MUST be able to:
   - Resume work after restart while offline
   - Re-authenticate using permitted offline mechanisms
   - Recover sessions without network dependency where authorized

4. **Human-Mediated Trust Support**
   The platform MUST support operational flows that rely on:
   - Face-to-face confirmation
   - Verbal agreements
   - Paper or verbal backups
   with post-facto reconciliation and auditability.

5. **Graceful Degradation**
   No core operation (sales, ticketing, check-in, inventory update) may hard-fail due to lack of connectivity.
   All failures MUST degrade gracefully into recoverable states.

6. **Field Override with Accountability**
   Authorized human overrides MUST exist for exceptional circumstances, with:
   - Mandatory justification
   - Immutable audit logs
   - Deferred reconciliation requirements

Field operations are not edge cases. They are the primary execution context for WebWaka.

## 3. Canonical Actor Hierarchy & Delegation Model

The WebWaka platform has a **five-level actor hierarchy** that is a foundational invariant. This model is designed to be enforceable by the system, not just a conceptual framework.

### 3.1. The Five Canonical Actor Classes

| Level | Actor Class | Description | Primary Relationship |
|:---:|:---|:---|:---|
| **L1** | **Super Admin** | The global owner and operator of the WebWaka platform. | Manages Partners |
| **L2** | **Partner** | A direct client of the Super Admin. May fully white-label the platform. | Manages Tenants |
| **L3** | **Tenant** | A client of a Partner (or directly of the Super Admin). Owns a platform instance. | Manages Vendors |
| **L4** | **Vendor** | A client of a Tenant. Sells products/services within a Tenant's marketplace. | Sells to End Users |
| **L5** | **End User** | The final consumer of products and services on a Tenant's platform. | Interacts with Vendors |

### 3.2. Authority & Delegation (Core Invariant)

**Nothing is hardcoded. Everything is configurable and inheritable.**

Higher-level actors have the authority to define roles, capabilities, permissions, pricing, and more for the actors below them. This creates a fully dynamic and delegable authority model.

**Example:**
- A **Partner (L2)** can define a set of subscription plans for their **Tenants (L3)**.
- A **Tenant (L3)** can then define specific roles and permissions for their **Vendors (L4)**.
- A **Vendor (L4)** can set prices for their products, which are then purchased by **End Users (L5)**.

### 3.3. Key Principles of the Actor Model

- **No Role Collision:** An actor cannot simultaneously be a Partner and a Tenant in the same hierarchy. The roles are mutually exclusive to prevent conflicts of interest.
- **Enforceable by Design:** The actor hierarchy is not just a naming convention; it is enforced by the platform's architecture and access control systems.
- **Recursive Authority:** The Super Admin uses the same tools to manage Partners that Partners use to manage Tenants, reinforcing the recursive platform design.

---

### 3.4. Canonical Sub-Roles Within Actor Classes (V4.1 Clarification)

Each actor class (L1-L5) can have internal sub-roles that define operational authority and economic participation. These sub-roles are **not new actor classes**—they are operational distinctions within a class.

#### 3.4.1. Sub-Role Taxonomy

| Actor Class | Sub-Role | Definition | Creation Authority | Role Assignment Authority | Commission Eligibility | Transaction Representation |
|:------------|:---------|:-----------|:-------------------|:-------------------------|:----------------------|:--------------------------|
| **L1: Super Admin** | Admin | Full platform authority | N/A (root) | Self | No | Yes |
| **L1: Super Admin** | Staff | Operational support for Super Admin | Super Admin | Super Admin | Optional (salaried) | Yes (delegated) |
| **L2: Partner** | Admin | Full authority within Partner scope | Super Admin | Self | Yes (markup share) | Yes |
| **L2: Partner** | Staff | Operational support for Partner | Partner Admin | Partner Admin | Optional (salaried) | Yes (delegated) |
| **L3: Tenant** | Admin | Full authority within Tenant scope | Partner or Super Admin | Self | Yes (platform fee share) | Yes |
| **L3: Tenant** | Staff | Operational support for Tenant | Tenant Admin | Tenant Admin | Optional (salaried) | Yes (delegated) |
| **L4: Vendor** | Admin | Full authority within Vendor scope | Tenant Admin | Self | Yes (transaction revenue) | Yes |
| **L4: Vendor** | Staff | Salaried employee of Vendor | Vendor Admin | Vendor Admin | No (salaried) | Yes (on behalf of Vendor) |
| **L4: Vendor** | Agent (Independent) | Commission-based seller for Vendor | Vendor Admin | Vendor Admin | Yes (commission-based) | Yes (on behalf of Vendor) |
| **L4: Vendor** | External Contractor | Third-party service provider | Vendor Admin | Vendor Admin | Yes (contract-based) | No (unless explicitly delegated) |
| **L5: End User** | Customer | Final consumer of products/services | Self-registration or Vendor | N/A | No | No |

#### 3.4.2. Sub-Role Creation Rules

- **Who Can Create Whom:**
  - Super Admin can create Partners
  - Partners can create Tenants
  - Tenants can create Vendors
  - Vendors can create Staff, Agents, and External Contractors
  - End Users self-register or are created by Vendors (e.g., for loyalty programs)

- **Who Can Assign Roles:**
  - Only the **Admin** sub-role within an actor class can assign sub-roles to other actors within the same class
  - Example: A Tenant Admin can assign "Staff" or "Agent" roles to users within the Tenant's scope
  - Sub-role assignments are auditable and must be logged

#### 3.4.3. Commission Eligibility Rules

- **Salaried Staff:** Do not earn commissions on transactions. They are compensated via fixed salary.
- **Independent Agents:** Earn commissions on every transaction they facilitate. Commission rates are defined by the Vendor Admin.
- **Admins:** May earn commissions based on their actor class (e.g., Partner Admins earn markup share, Tenant Admins earn platform fee share).
- **External Contractors:** Earn fees based on contractual agreements, not transaction commissions.

#### 3.4.4. Transaction Representation Rules

- **Who Can Represent Whom:**
  - Staff and Agents can transact **on behalf of** their parent actor (e.g., a Vendor Staff member can sell products on behalf of the Vendor)
  - External Contractors cannot represent their parent actor in transactions unless explicitly delegated via a formal contract
  - All transactions must record the **acting sub-role** for audit and commission attribution

#### 3.4.5. Canonical Use Cases

**Use Case 1: POS Staff**
- A Vendor Admin creates a "Staff" sub-role for an employee
- The Staff member uses the POS to sell products on behalf of the Vendor
- The Staff member does not earn commissions; they are salaried
- All sales are attributed to the Vendor, not the Staff member

**Use Case 2: Marketplace Sales Agent**
- A Vendor Admin creates an "Agent (Independent)" sub-role for a commission-based seller
- The Agent uses the marketplace to sell products on behalf of the Vendor
- The Agent earns a commission on every sale (e.g., 5% of transaction value)
- All sales are attributed to the Vendor, with commission split recorded for the Agent

**Use Case 3: Transport Ticket Seller**
- A Tenant Admin (Motor Park) creates an "Agent (Independent)" sub-role for an independent ticket seller
- The Agent sells tickets on behalf of the Motor Park
- The Agent earns a commission on every ticket sold (e.g., ₦50 per ticket)
- Seat inventory is updated in real-time across all sales channels

**Use Case 4: Independent vs Salaried Actors**
- **Salaried:** A Vendor hires a Staff member with a fixed monthly salary of ₦100,000. The Staff member does not earn commissions.
- **Independent:** A Vendor onboards an Agent with a 10% commission rate. The Agent earns ₦10,000 on every ₦100,000 in sales they facilitate.

### 3F. Cross-Suite Identity & Contextual Role Binding

A single human identity MAY participate in multiple suites, tenants, and operational contexts simultaneously. Identity within WebWaka is unified at the human level but contextually bound at the role level.

#### Identity Binding Rules

   A single human MAY concurrently act as:
   - A Vendor Admin in Commerce
   - An Agent in Transportation
   - A Staff member in Hospitality
   - An End User in another tenant

2. **Context-Scoped Authority**
   Roles, permissions, economic rights, and obligations are strictly scoped to the active context.

3. **Contextual Economic Attribution**
   All transactions, commissions, incentives, and liabilities MUST be attributed to:
   - The active role
   - The active tenant or suite
   - The active authority chain

4. **Explicit Context Switching**
   Context changes MUST be:
   - Explicit
   - Logged
   - Visible to the user
   especially in offline scenarios.

5. **Audit-Safe Role Overlap**
   The platform MUST maintain unambiguous audit trails even when a single human occupies multiple roles across contexts.

Identity is global. Authority is contextual. Economics are scoped.

## 4. Economic & Affiliate Graph (NEW — CRITICAL)

The WebWaka platform is designed to support complex, multi-level economic relationships. This section defines the canonical Economic & Affiliate Graph, a non-negotiable model for all revenue sharing, commissions, and remuneration.

### 4.1. Core Principles

- **No Hardcoded Economics:** All economic relationships are configurable and inheritable.
- **Separation of Concerns:** Subscription revenue is distinct from transactional revenue.
- **Delegable Authority:** Higher-level actors can define economic models for lower-level actors.
- **Auditability:** All economic events are auditable and traceable.

### 4.2. Revenue Sharing Chains

Revenue sharing flows down the actor hierarchy:

**Super Admin (L1) → Partner (L2) → Tenant (L3) → Vendor (L4) → Agent (L5)**

- The **Super Admin** sets the base pricing for the platform.
- **Partners** can mark up pricing for their **Tenants**.
- **Tenants** can charge **Vendors** subscription fees and/or transaction commissions.
- **Vendors** can employ **Agents** and define their commission structures.

### 4.3. Commission Inheritance & Overrides

- **Inheritance:** Economic models are inherited down the hierarchy unless explicitly overridden.
- **Overrides:** Higher-level actors can set limits on the overrides that lower-level actors can apply.

### 4.4. Subscription vs. Transactional Revenue

- **Subscription Revenue:** Recurring fees for access to the platform and its suites.
- **Transactional Revenue:** Commissions and fees charged on a per-transaction basis.

### 4.5. Staff vs. Independent Agent Remuneration

- **Staff Agents:** Salaried employees of a Vendor or Tenant.
- **Independent Agents:** Commission-based agents who are not employees.

### 4.6. Contractual Boundaries

- Each level of the hierarchy operates under a distinct contractual agreement.
- The platform must support the creation and management of these contracts.

### 4.7. End-to-End Transaction Narrative (V4.1 Clarification)

This section provides a concrete, single-transaction walkthrough to illustrate how economic value flows through the WebWaka platform across all actor levels.

#### 4.7.1. Transaction Scenario

**Context:**
- A **Partner** (L2) operates a white-labelled platform for SMEs in Nigeria
- A **Tenant** (L3) is a fashion retailer using the Multi-Vendor Marketplace suite
- A **Vendor** (L4) is a shoe seller on the Tenant's marketplace
- An **Agent** (L4 sub-role) is an independent sales agent working for the Vendor
- An **End User** (L5) is a customer purchasing shoes

**Transaction:**
- End User purchases a pair of shoes for **₦20,000**
- The Agent facilitated the sale via a referral link

#### 4.7.2. Economic Flow Breakdown

| Step | Actor | Action | Amount | Running Total |
|:-----|:------|:-------|:------:|:-------------:|
| **1** | **End User (L5)** | Pays ₦20,000 for shoes | -₦20,000 | ₦0 |
| **2** | **Platform** | Receives payment and holds in escrow | +₦20,000 | ₦20,000 |
| **3** | **Agent (L4 sub-role)** | Earns 5% commission (pre-configured by Vendor) | +₦1,000 | ₦19,000 |
| **4** | **Vendor (L4)** | Earns net revenue after Agent commission | +₦14,000 | ₦5,000 |
| **5** | **Tenant (L3)** | Charges 20% platform fee on Vendor revenue | +₦3,000 | ₦2,000 |
| **6** | **Partner (L2)** | Charges 40% markup on Tenant platform fee | +₦1,200 | ₦800 |
| **7** | **Super Admin (L1)** | Receives base platform fee (remainder) | +₦800 | ₦0 |

#### 4.7.3. Economic Flow Summary

**Total Transaction Value:** ₦20,000

**Distribution:**
- **Agent (L4 sub-role):** ₦1,000 (5% of transaction)
- **Vendor (L4):** ₦14,000 (70% of transaction, after Agent commission)
- **Tenant (L3):** ₦3,000 (15% of transaction, platform fee)
- **Partner (L2):** ₦1,200 (6% of transaction, markup on Tenant fee)
- **Super Admin (L1):** ₦800 (4% of transaction, base platform fee)

**Key Observations:**
1. The **Agent** earns a commission on the gross transaction value (before platform fees)
2. The **Vendor** earns the majority of the transaction value (70%)
3. The **Tenant** earns a platform fee on the Vendor's revenue
4. The **Partner** earns a markup on the Tenant's platform fee
5. The **Super Admin** earns the base platform fee (the remainder after all other deductions)

#### 4.7.4. Revenue Routing Rules

**Rule 1: Commission Precedence**
- Agent commissions are deducted **first**, before any platform fees
- This ensures that sales agents are always compensated, regardless of platform fee structures

**Rule 2: Platform Fee Inheritance**
- Platform fees flow **upward** through the actor hierarchy (Tenant → Partner → Super Admin)
- Each level can define their own fee structure, subject to limits set by the level above

**Rule 3: Markup Limits**
- Partners can set markup limits for their Tenants (e.g., "Tenants cannot charge Vendors more than 25% platform fee")
- Tenants can set commission limits for their Vendors (e.g., "Vendors cannot offer Agents more than 10% commission")

**Rule 4: Auditability**
- Every transaction must record:
  - Gross transaction value
  - Agent commission (if applicable)
  - Vendor net revenue
  - Tenant platform fee
  - Partner markup
  - Super Admin base fee
- All economic events are immutable and auditable

#### 4.7.5. Alternative Scenarios

**Scenario A: No Agent Involved**
- If the transaction is completed without an Agent, the Agent commission (₦1,000) is added to the Vendor's net revenue
- New Vendor revenue: ₦15,000 (instead of ₦14,000)

**Scenario B: Direct Tenant (No Partner)**
- If the Tenant is a direct client of the Super Admin (no Partner intermediary), the Partner markup (₦1,200) is added to the Super Admin's base fee
- New Super Admin fee: ₦2,000 (instead of ₦800)

**Scenario C: Subscription-Only Model**
- If the Tenant uses a subscription-only model (no transaction fees), the Vendor pays a fixed monthly fee (e.g., ₦50,000/month) instead of per-transaction fees
- In this case, the economic flow is:
  - Vendor pays ₦50,000/month to Tenant
  - Tenant pays ₦20,000/month to Partner (40% markup)
  - Partner pays ₦12,000/month to Super Admin (60% of Partner fee)
  - Super Admin retains ₦12,000/month as base platform fee

## 5. Role–Capability–Permission–Pricing Meta-Model (NEW)

The WebWaka platform is governed by a canonical meta-model that defines the relationships between roles, capabilities, permissions, and pricing. This model, known as the **WebWaka Entitlement & Economic Graph (WEEG)**, is a non-negotiable abstraction that all suites and features MUST use.

### 5.1. Core Components of the WEEG

- **Roles:** Named sets of capabilities and permissions (e.g., "Tenant Admin," "Vendor Staff").
- **Capabilities:** High-level features or functions (e.g., "Create Marketplace," "Manage Inventory").
- **Permissions:** Granular access rights (e.g., "can_delete_product," "can_view_reports").
- **Pricing Plans:** Bundles of capabilities and permissions with associated costs.
- **Benefits:** Non-functional entitlements (e.g., "24/7 Support," "Dedicated Account Manager").
- **Contracts:** Formal agreements that bind an actor to a specific pricing plan and set of benefits.

### 5.2. Inheritance and Overrides

- **Inheritance:** The WEEG supports multi-level inheritance. For example, a "Tenant Admin" role can inherit all the capabilities and permissions of a "Vendor Staff" role, plus additional administrative capabilities.
- **Overrides:** Higher-level actors can override the pricing and benefits of lower-level actors, within defined limits.

### 5.3. Auditability and Delegation

- **Auditability:** All changes to the WEEG are auditable. The system must record who made the change, when it was made, and what was changed.
- **Delegation:** Higher-level actors can delegate the management of roles, capabilities, and permissions to lower-level actors.

## 8. Inventory, Eventing & Cross-Suite Synchronisation

### 8.1. Event-Driven Architecture as Core Invariant

- Audit trails for compliance

### 8.2. Inventory Synchronization

### 9.4. AI-Native Enforcement Invariant (V4.1 Clarification)

**HARD INVARIANT (Non-Negotiable):**

**No feature may ship without an AI-extension path, even if initially disabled.**

This is a foundational architectural rule that applies to **all features, modules, and suites** on the WebWaka platform. It is enforced by CI/CD and cannot be bypassed.

#### 9.4.1. What This Means

Every feature, module, or suite must be designed with an **AI-extension path** from Day 1. This does not mean that AI functionality must be implemented immediately, but the architecture must support AI integration without requiring a refactor.

**Examples of AI-Extension Paths:**

| Feature | AI-Extension Path | Initial State | Future AI Enhancement |
|:--------|:------------------|:--------------|:----------------------|
| **Product Catalog** | Product descriptions can be AI-generated | Manual entry only | AI generates descriptions from product images and attributes |
| **Inventory Management** | Inventory forecasting can be AI-powered | Manual reorder points | AI predicts reorder points based on sales trends |
| **Customer Support** | Support tickets can be AI-triaged | Manual triage by staff | AI categorizes and routes tickets automatically |
| **Route Management** | Route optimization can be AI-driven | Manual route planning | AI optimizes routes based on traffic, weather, and demand |
| **Pricing** | Dynamic pricing can be AI-adjusted | Fixed pricing rules | AI adjusts prices based on demand, competition, and inventory |

#### 9.4.2. Why This Is Non-Negotiable

**Reason 1: Prevents Technical Debt**
- If features are built without AI-extension paths, adding AI later requires refactoring
- Refactoring is expensive, error-prone, and delays AI adoption
- By requiring AI-extension paths from Day 1, we prevent this technical debt

**Reason 2: Ensures Competitive Advantage**
- AI is a horizontal capability that differentiates WebWaka from competitors
- If AI is bolted on after the fact, it will always feel like an afterthought
- By designing for AI from Day 1, we ensure that AI is seamlessly integrated into every feature

**Reason 3: Supports Tenant Flexibility**
- Some Tenants will want to use AI immediately, while others will not
- By requiring AI-extension paths, we ensure that Tenants can enable AI when they are ready
- This supports a gradual adoption model, where Tenants can start with manual processes and upgrade to AI over time

#### 9.4.3. How This Is Enforced

**CI/CD Check:**
- Every PR that introduces a new feature, module, or suite must include an AI-extension path in the design document
- The CI/CD pipeline will reject any PR that does not include this documentation
- The design document must specify:
  - What AI capabilities could enhance this feature in the future
  - What data structures or APIs are required to support AI integration
  - What permissions or quotas are required to enable AI for this feature

**Architecture Review:**
- The Chief Architect reviews all design documents to ensure that AI-extension paths are realistic and feasible
- If the AI-extension path is not clear, the PR is rejected and the developer must revise the design

**Example CI/CD Check:**
```yaml
# .github/workflows/ai-extension-check.yml
name: AI Extension Path Check
on: [pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - name: Check for AI Extension Path
        run: |
          if ! grep -q "AI-Extension Path" docs/design/*.md; then
            echo "ERROR: No AI-extension path documented for this feature"
            exit 1
          fi
```

#### 9.4.4. What This Does NOT Mean

**This invariant does NOT require:**
- Implementing AI functionality in the first version of a feature
- Training custom AI models for every feature
- Hiring AI specialists for every development team

**This invariant DOES require:**
- Thinking about AI integration during the design phase
- Structuring data and APIs to support AI in the future
- Documenting the AI-extension path in the design document

#### 9.4.5. Canonical Examples

**Example 1: Product Catalog (Commerce Suite)**
- **Initial Implementation:** Tenants manually enter product names, descriptions, and prices
- **AI-Extension Path:** The product catalog API includes a `description_source` field that can be set to `manual` or `ai_generated`
- **Future Enhancement:** Tenants can enable AI-generated descriptions by setting `description_source` to `ai_generated` and providing product images and attributes

**Example 2: Route Management (Transportation Suite)**
- **Initial Implementation:** Tenants manually define routes, schedules, and vehicle assignments
- **AI-Extension Path:** The route management API includes an `optimization_mode` field that can be set to `manual` or `ai_optimized`
- **Future Enhancement:** Tenants can enable AI-optimized routes by setting `optimization_mode` to `ai_optimized` and providing traffic data, weather data, and demand forecasts

**Example 3: Customer Support (All Suites)**
- **Initial Implementation:** Support tickets are manually triaged by staff
- **AI-Extension Path:** The support ticket API includes a `triage_mode` field that can be set to `manual` or `ai_triaged`
- **Future Enhancement:** Tenants can enable AI-triaged tickets by setting `triage_mode` to `ai_triaged` and providing historical ticket data for training

### 9E. Human-First Fallback Doctrine for AI

AI is native to WebWaka, but AI MUST NEVER be a single point of failure.

#### Mandatory AI Fallback Rules

1. **No Hard Dependency**
   No critical user flow (payments, ticketing, POS, access control) may require AI availability to function.

2. **Deterministic Alternatives**
   Every AI-assisted operation MUST have:
   - A deterministic non-AI fallback
   - A manual override path where permitted

3. **Advisory vs Authoritative AI**
   In regulated or sensitive domains (finance, health, identity):
   - AI outputs are advisory by default
   - Explicit approval is required for AI-driven authority

4. **Cost & Quota Awareness**
   AI usage MUST respect:
   - User-level quotas
   - Tenant-level budgets
   - Platform-level safeguards

5. **Explainability & Override**
   AI outputs MUST be explainable and overridable by authorized humans.

AI augments human capability. It does not replace human responsibility.

#### 12.7.2. Integration Levels Explained

**Platform-Level Integrations:**
- Installed and configured by the **Super Admin** only
- Available to all Partners, Tenants, and Vendors (unless explicitly restricted)
- Examples: AWS S3, Stripe, Twilio, SendGrid
- Use case: Core infrastructure integrations that all actors rely on

**Partner-Level Integrations:**
- Installed and configured by the **Partner**
- Available to all Tenants and Vendors under that Partner (unless explicitly restricted)
- Examples: Partner-specific payment gateway, Partner-specific SMS provider
- Use case: A Partner wants to use a regional payment gateway (e.g., Paystack for Nigeria) instead of the global default (Stripe)

**Tenant-Level Integrations:**
- Installed and configured by the **Tenant**
- Available to all Vendors under that Tenant (unless explicitly restricted)
- Examples: Tenant-specific CRM (e.g., HubSpot), Tenant-specific analytics tool (e.g., Google Analytics)
- Use case: A Tenant wants to integrate their marketplace with their existing CRM

**Vendor-Level Integrations:**
- Installed and configured by the **Vendor**
- Available only to that Vendor
- Examples: Vendor-specific accounting software (e.g., QuickBooks), Vendor-specific inventory management tool
- Use case: A Vendor wants to sync their WebWaka inventory with their existing accounting system

#### 12.7.3. Authority Rules

**Rule 1: Higher Levels Can Override Lower Levels**
- The Super Admin can override any integration configuration at any level
- A Partner can override any integration configuration for their Tenants and Vendors
- A Tenant can override any integration configuration for their Vendors
- Vendors cannot override configurations set by higher levels

**Rule 2: Higher Levels Can Restrict Lower Levels**
- The Super Admin can restrict which integrations Partners can install
- A Partner can restrict which integrations Tenants can install
- A Tenant can restrict which integrations Vendors can install
- Example: A Partner may disallow Tenants from installing third-party payment gateways to enforce a single payment provider

**Rule 3: Kill-Switch Authority**
- The Super Admin can disable any integration at any level (global kill-switch)
- A Partner can disable any integration for their Tenants and Vendors (Partner kill-switch)
- A Tenant can disable any integration for their Vendors (Tenant kill-switch)
- Vendors cannot invoke kill-switches
- Use case: If an integration is compromised or malfunctioning, higher-level actors can immediately disable it

**Rule 4: Audit Logs Are Hierarchical**
- The Super Admin can view audit logs for all integrations at all levels
- A Partner can view audit logs for integrations at the Partner, Tenant, and Vendor levels (but not other Partners)
- A Tenant can view audit logs for integrations at the Tenant and Vendor levels (but not other Tenants)
- A Vendor can view audit logs for their own integrations only

#### 12.7.4. Canonical Use Cases

**Use Case 1: Super Admin Installs Platform-Level Integration**
- The Super Admin installs Stripe as the default payment gateway for the entire platform
- All Partners, Tenants, and Vendors can use Stripe for payments
- The Super Admin configures the Stripe API keys and webhook endpoints

**Use Case 2: Partner Installs Partner-Level Integration**
- A Partner in Nigeria installs Paystack as an alternative payment gateway
- All Tenants and Vendors under that Partner can use Paystack instead of Stripe
- The Partner configures the Paystack API keys and webhook endpoints

**Use Case 3: Tenant Installs Tenant-Level Integration**
- A Tenant installs Google Analytics to track customer behavior on their marketplace
- All Vendors under that Tenant benefit from the analytics data
- The Tenant configures the Google Analytics tracking ID

**Use Case 4: Vendor Installs Vendor-Level Integration**
- A Vendor installs QuickBooks to sync their sales data with their accounting system
- Only that Vendor has access to the QuickBooks integration
- The Vendor configures the QuickBooks API keys and sync schedule

**Use Case 5: Super Admin Invokes Kill-Switch**
- A security vulnerability is discovered in a third-party integration (e.g., a payment gateway)
- The Super Admin immediately disables the integration across the entire platform
- All transactions using that integration are halted until the vulnerability is patched

**Use Case 6: Tenant Restricts Vendor Integrations**
- A Tenant wants to enforce a single payment gateway for all Vendors on their marketplace
- The Tenant disables the ability for Vendors to install alternative payment gateways
- Vendors can only use the payment gateway configured by the Tenant

| **Linting** | Ensures that all code follows style guidelines | Rejects PRs with linting errors (e.g., ESLint, Prettier) |
| **Security Scans** | Ensures that all code is free of known vulnerabilities | Rejects PRs with high-severity security issues (e.g., Snyk, Dependabot) |

#### 16.4.4. Why This Distinction Matters

**For Governance:**
- Governance CI ensures that the WebWaka institutional model is preserved across all repositories
- Without Governance CI, developers could accidentally (or intentionally) introduce architectural drift
- Governance CI is **repository-agnostic**—it applies to all repositories in the webwaka-factory organization

**For Product Development:**
- Product CI ensures that code quality and functionality standards are met
- Without Product CI, bugs and regressions could be introduced into production
- Product CI is **repository-specific**—each repository has its own test suite and build process

**For Rebuild:**
- When rebuilding WebWaka in a WebWakaHub GitHub organization, Governance CI must be set up **first**
- Governance CI ensures that the new organization follows the same institutional rules as the original
- Product CI can be set up incrementally as each repository is created

#### 16.4.5. Governance CI Precedence

**Rule: Governance CI failures take precedence over Product CI failures.**

If a PR fails Governance CI, it **cannot be merged**, even if Product CI passes. This ensures that institutional integrity is never compromised for the sake of shipping features quickly.

**Example:**
- A developer submits a PR that adds a new feature to the Commerce Suite
- Product CI passes (all tests pass, code builds successfully)
- Governance CI fails (the feature does not include an AI-extension path)
- **Result:** The PR is rejected and the developer must add an AI-extension path before resubmitting

When rebuilding WebWaka in a WebWakaHub GitHub organization, the following Governance CI checks must be implemented **before any product code is written**:

### 17.6. Explicit Discards (V4.1 Clarification)

This section lists what is **intentionally NOT carried forward** in a zero-based reconstruction of WebWaka. These discards are critical to prevent contamination in the clean-slate rebuild.

#### 17.6.1. What Is Discarded

| Category | What Is Discarded | Why | What Replaces It |

| **Legacy Integrations** | All integrations that are not documented in the canonical integration registry | Legacy integrations may be insecure, unmaintained, or incompatible with the new architecture | All integrations are re-evaluated and re-implemented using the canonical integration subsystem |

| **Deprecated APIs** | All APIs that are not documented in the canonical API registry | Deprecated APIs may be insecure, unmaintained, or incompatible with the new architecture | All APIs are re-evaluated and re-implemented using the canonical API design |

#### 17.6.2. Why Discards Are Non-Negotiable

**Reason 1: Prevent Contamination**
- Legacy systems accumulate technical debt, security vulnerabilities, and architectural inconsistencies over time
- Carrying forward legacy artifacts contaminates the new system and undermines the clean-slate rebuild
- By explicitly discarding legacy artifacts, we ensure that the new system is built on a solid foundation

**Reason 2: Enforce Institutional Discipline**
- A zero-based reconstruction is an opportunity to enforce institutional discipline
- By discarding all undocumented, implicit, and ad-hoc artifacts, we force the team to document everything
- This creates a culture of transparency, accountability, and knowledge sharing

**Reason 3: Support Long-Term Maintainability**
- Systems that carry forward legacy artifacts are difficult to maintain and evolve
- By discarding legacy artifacts, we ensure that the new system is maintainable for 10+ years
- This supports the long-term vision of WebWaka as a platform-for-platforms
